# RARA PC軽量化ツール

移行のお知らせ

操作は統合ウィンドウへ移行しました。メニューの RARA > アバター軽量化・Quest・iOS対応ツール を選ぶと統合ウィンドウが開き、上部のチップで対象を PC(または Quest・両方)に切り替えられます(最適化エンジンは同じものを共有しています)。この単体UI(PCOptimizerWindow)はメニューからは外しましたが、クラスと Open() API はコード互換のために残しています。

VRChat の PC(Windows)用アバターの性能ランクを PC 基準で引き上げる Unity Editor 拡張。
元のアバターは一切変更しない非破壊方式で、「アバター名_Opt」の複製と、同じ内容のプレファブアセットを新しく生成する。

## 概要

RARA のアバター軽量化ツールは対象ユーザーで役割を分けている。

- ざっくり Quest / スマホ対応にしたい人 → 姉妹ツール RARA Quest対応コンバーター(シェーダー変換・テクスチャベイクまで行う)
- PC のまま使いたいが、Very Poor を避けて Poor 以上に載せたい人 → このツール
- PC で Good まで狙いたい人 → このツール

つまり本ツールは、Quest 変換ではなく PC(Windows)基準でのランク改善に特化している。ポリゴンを減らす機能やシェーダーを変換する機能は持たず、テクスチャメモリ・マテリアルスロット・メッシュ・PhysBone といった、PC でランクを押し下げている要素を雑にまとめて削減するのが役目。

主な特徴

- 非破壊。元アバターは触らず、「アバター名_Opt」のシーン複製とプレファブアセットが新しく出現する形式
- テクスチャメモリ削減、マテリアル統合(テクスチャ無しの色だけ材質は色セルをベイクして統合)、エミッションマスク生成、トグル固定、PhysBone マージを1つのウィンドウでまとめて実行
- 診断で現在の PC ランクと目標ランクの閾値を常に表示しながら作業できる
- 仕上げに「Quest対応コンバーターへ引き継ぐ」で、生成した _Opt をそのまま Quest / スマホ対応の変換フローへ渡せる

## 導入条件

- Unity 2022.3 系(VRChat 公式指定バージョン。2022.3.22f1 で動作確認)
- VRChat Avatars SDK(VCC / ALCOM で作成したアバタープロジェクト。3.10.4 で動作確認)
- RARA Quest対応コンバーター本体が同梱必須
  - 本ツールは、トグル整理・PhysBone マージ・アトラス統合・テクスチャベイク・生成物の持続化(GUID 保持の上書き保存)といった共通エンジンを Quest対応コンバーターのコードから参照している。PC軽量化ツール単体では動かない
  - そのため導入・エクスポートは Assets/RARA フォルダごと(QuestConverter と PCOptimizer の両方)まとめて扱うこと
- AvatarOptimizer(AAO)の事前導入を強く推奨(1.9 系で動作確認)
  - メッシュ結合そのものはビルド時に AAO が担当する分業設計。トグルを固定してもスロットやスキンメッシュがまとまるのはビルド時なので、AAO の Trace and Optimize が無いと削減効果が出にくい。実質必須と考えてよい
  - 導入は VCC/ALCOM にリポジトリを追加して行う。ブラウザで https://vpm.anatawa12.com/add-repo を開き、VCC/ALCOM が開いたら Add Repository → プロジェクトに AvatarOptimizer を追加 → Unity へ戻る
- レンダーパイプラインは Built-in のみ対応
- lilToon は必須ではない(PC ではシェーダーを変換しないため。元マテリアルはそのまま使う)

## 導入方法

VCC / ALCOM から VPM パッケージ(jp.rara.avatar-tools)として導入する。詳しい手順は
リポジトリ直下の README.md「導入(VCC / ALCOM)」を参照。パッケージは
Packages/jp.rara.avatar-tools/ 以下に配置され、コンパイルが終わるとメニューバーに
RARA > アバター軽量化・Quest・iOS対応ツール が追加される(統合ウィンドウ。上部のチップで対象を PC / Quest / 両方に切り替える)。この PC軽量化ツール単体UI(PCOptimizerWindow)はメニューからは外れており、統合ウィンドウで PC を選ぶと同じ最適化が行える。

追加の設定は不要。アンインストールは VCC / ALCOM の Manage Project からパッケージを外すだけでよい。

## 想定フロー

このツールはポリゴン削減はできない。PC でランクを上げる前提として、まず外部ツールでポリゴンを減らしておく。

1. 外部の減面(ポリゴン削減)ツールで、まずポリゴンを 70,000 以下にする
   - PC では 70,000 を超えると一気に Very Poor になる。Poor / Medium / Good のいずれを狙うにも 70,000 以下が前提
   - Good を狙う場合も 70,000 以下でよい(ポリゴンだけで見れば 32,001〜70,000 は Good に載る)。Excellent まで詰めるなら 32,000 以下
2. 本ツールで、テクスチャメモリ・マテリアルスロット・メッシュ・PhysBone を削減して目標ランクへ寄せる(雑にアバターを軽量化する)
3. 必要なら「Quest対応コンバーターへ引き継ぐ」で、生成した _Opt を Quest / スマホ対応の変換フローへ渡す

## 使い方

ウィンドウは上から順に進めば完了する構成になっている。

1. シーンに置いたアバター(VRCAvatarDescriptor 付き)をウィンドウで選択する(選択すると自動で診断が走る)
2. PC適合診断で、現在の総合ランク・カテゴリ別ランクと、目標ランクの閾値を確認する
3. 目標ランクを選ぶ(既定は Good。Excellent も選べる)。以降の各セクションの提案がこの目標に合わせて表示される
4. テクスチャメモリ削減: 寄与の大きいテクスチャから縮小提案が並ぶ。個別「適用」または「提案をすべて適用」で縮小計画に登録する。目標ランクの上限に収まるよう自動調整もできる
5. マテリアル統合: 互換性のあるマテリアルを1枚のアトラスにまとめてスロット数を減らす。テクスチャ無しの色だけ材質は色セルを、エミッションはエミッションマスクセルをアトラスにベイクして、統合後も色・発光が保たれる
6. トグル整理: トグルで切り替える衣装・アクセサリを検出。各グループを「トグル維持 / 表示で固定 / 非表示で固定」から選ぶ。固定するとビルド時に AAO が結合できるようになり、スキンメッシュ数・スロット数が減る
7. PhysBone 整理: 設定が一致する兄弟 PhysBone チェーンをマージしてコンポーネント数を削減する(揺れは維持)。稼働させる PhysBone の取捨選択や、積極マージも設定できる
8. メッシュ削減(AAO連携): 服の下に隠れて見えない肌などを、shrink ブレンドシェイプ検出や AAO の Trace and Optimize でビルド時に削減する
9. 実行: 「PC軽量化版を生成」で、元アバターを触らずに _Opt 複製とプレファブを生成する。レポートで結果を確認する
10. Quest対応へ引き継ぐ: 仕上げに Quest / スマホ対応が必要なら、生成した _Opt をワンクリックで Quest対応コンバーターへ渡す

## できること

1. PC適合診断
   - VRChat SDK の性能統計 API を Windows(PC)基準で実行し、総合ランクとカテゴリ別ランクを表示する
   - 目標ランク(既定 Good)の閾値を各カテゴリに併記し、どのカテゴリが目標を超えているかを一目で分かるようにする
   - EditorOnly タグ付きのオブジェクトは、実際のビルドと同様に除外した状態で計算する
2. テクスチャメモリ削減
   - 全テクスチャ(メイン・エミッション・ノーマル・マスク類)のメモリ寄与を推定し、寄与の大きいものから縮小提案を並べる
   - 提案は「適用」(個別)/「提案をすべて適用」(一括)で縮小計画に登録できる。目標ランクのテクスチャメモリ上限に収まるよう自動で計画を作ることもできる
   - 縮小計画に登録されたテクスチャは、生成時に低解像度のコピーが Generated に作られ、_Opt がそれを参照する。元のテクスチャとそのインポート設定には一切触れない
3. マテリアル統合と色だけ材質の色ベイク
   - 互換性のある複数マテリアルを1枚のアトラステクスチャ+1マテリアルに統合し、メッシュの UV を再マップしてスロット数を減らす
   - テクスチャを持たない色だけの材質(単色マテリアル)も統合対象にする。マテリアルの色を拾って単色セルとしてアトラスにベイクするので、統合後も各パーツの色が保たれる(見た目はある程度の近似)
   - エミッション色/エミッションマップは、エミッション用アトラスのセルにベイクして発光を引き継ぐ
   - NonToon(jp.lilxyzw)対応: メインテクスチャ(_BaseTexture)・ノーマル(_NormalMap)を系統別に解決してアトラス化する。統合は同じ NonToon 同士のみで、lilToon など他系統とは混ざらない。NonToon はメインカラー(色乗算)とエミッションを持たないため、色はテクスチャそのまま・発光ベイクは行わない
   - マットキャップ(lilToon の映り込み)を持つマテリアルは、統合時にマットキャップを外す。lilToon のマットキャップは UV0 でマスク(_MatCapBlendMask 等)を参照するため、アトラスの UV 再配置でマスクが別セルを指し、代表プロパティへの統一とあいまって全面が金属マットキャップで白飛びするのを防ぐ。映り込みを残したいマテリアルは「アトラス除外」に指定すれば統合対象から外れ、マットキャップも保たれる。外したマットキャップは変換レポートに一覧表示され、統合プレビューにも「マットキャップは外れます」と表示される。マットキャップは分類の要因にしないため、異なるマットキャップを持つマテリアル同士も同じアトラスへ統合できる(統合効率が上がる)
   - アニメーションでマテリアル差し替えされるもの、UV が 0..1 範囲外(タイリング使用)のものは自動で対象外になる
   - カリング差を無視(既定オン): 片面(Cull Back)と両面(Cull Off)の違いを無視して統合する。カリングが混在したグループは、統合後のマテリアルが両面描画(Cull Off)になる。裏面が見えるようになるまれな見た目変化と、フィルレートの微増があるが、その分スロットは減りやすくなる。カリングが揃っているグループは元の値を保つ
   - アウトライン(輪郭線)の扱い(既定: アウトライン統合しない): アウトライン付き(例: Hidden/lilToonOutline)とアウトライン無し(例: lilToon)の統合方法をドロップダウンで選ぶ。しない=アウトライン有無が違えば別グループのまま。アウトラインを外して統合(推奨)=両者を統合し、統合マテリアルをプレーンlilToonへ揃える。服の輪郭線は消えるが、瞳・まつげ・眉・顔など黒縁として出やすい部分に不要なアウトラインが付かない。アウトライン付きに統一=アウトライン付き側へ揃える(輪郭の無かったマテリアルにアウトラインが付く。旧動作)。このモードでは瞳・顔系(eye / 瞳 / hitomi / pupil / iris / eyelash / まつげ / 睫 / eyebrow / 眉 / face / 顔 を名前に含むプレーンlilToon)を自動でアウトライン付与から回避し、プレーンのまま単独維持する。いずれのモードもファー・透過・カットアウトなどの他系統とは決して統合しない(アウトライン有無のペアだけを同一視する)
   - 統合予定のグループと、統合できないマテリアルの理由(シェーダー系統・カリング・アウトライン・アニメ差し替え・タイリング等)は、実行前にウィンドウ上でプレビュー表示される。オプションを切り替えれば統合できるようになるものは、その行にその場で案内が出る
4. トグル固定によるメッシュ・スロット削減
   - トグルで切り替える衣装・アクセサリを検出して一覧表示する。各グループを「トグル維持 / 表示で固定 / 非表示で固定」から選べる
   - 「表示で固定」= 常時表示にしてトグルアニメを外す。AAO がビルド時に結合できるようになり、スキンメッシュ数・スロット数を削減できる
   - 「非表示で固定」= そのメッシュを削除(EditorOnly)する
   - 元アバターの共有アニメクリップは変更しない(クローン側のクリップだけ編集)。m_IsActive 以外のアニメ(ブレンドシェイプ等)は維持される
5. PhysBone マージ
   - 設定が一致する兄弟 PhysBone チェーンを共通の親でまとめ、揺れを維持したままコンポーネント数を削減する(既定オン)
   - 積極マージ(既定オン)では、設定が完全一致しない兄弟チェーンも先頭の設定に統一してマージする(アニメ制御・パラメータ使用などの安全ゲートは維持)
   - 稼働させる PhysBone を選ぶ / 特定のものを削除する、といった取捨選択もできる。変換後のコンポーネント数はランク別上限ラダー(Excellent 4 / Good 8 / Medium 16 / Poor 32)と次ランクまであと何個削ればよいかの目安つきで色付き表示される(PC は超過してもアップロード可能でランクが下がるだけ)
6. メッシュ削減(AAO連携)
   - shrink(縮小)ブレンドシェイプを自動検出し、AAO の RemoveMeshByBlendShape を複製に自動セットアップできる(見えない部分だけをビルド時に削除)
   - Trace and Optimize が無ければ複製に追加してビルド時最適化を有効化する
   - 顔の表情・MMD・リップシンク用ブレンドシェイプは誤検出しない安全策付き
   - SkinnedMesh統合(顔以外を1つへ): 「メッシュ・トグル整理」セクションの先頭にある。顔(ビセーム/まばたき)以外の SkinnedMeshRenderer を AAO の Merge Skinned Mesh で 1 つへ統合し、SkinnedMesh 数とマテリアルスロット数を減らす(PC Good 上限 SkinnedMesh 2 / Excellent 上限 1 対策)。「統合しない / 顔以外を統合(推奨)」から選ぶ。既定は統合(旧設定から引き継いだ場合は統合しない=従来動作のまま)
   - 統合するもの: 顔以外の通常の SkinnedMeshRenderer(素体・衣装・髪・アクセサリ等)。統合しないもの: 顔メッシュ(ディスクリプタの VisemeSkinnedMesh / まばたきメッシュ=分離維持)、Cloth が付いたメッシュ、EditorOnly、プレビューで個別に「統合しない」を選んだメッシュ、多重描画(マテリアルスロット数>サブメッシュ数。統合するとビルド時にポリゴンが増えるため自動除外。1.7.1〜)、統合対象間で material.(マテリアルプロパティ)アニメーションの対象が食い違うメッシュ(統合すると統合先全体へ波及するため自動除外。1.7.1〜)
   - 表情・アニメは保たれる: 顔メッシュは分離維持でビセーム・まばたき・表情がそのまま動く。顔以外に残るブレンドシェイプ・シェイプ/マテリアルアニメも、統合時に AAO がブレンドシェイプ改名・マテリアルスロット再マップ・アニメの再パスを自動で行うため、追従して動き続ける(同名衝突は別名化で回避)。このためこのツール側で追加のアニメ書き換えは不要
   - ただし表示/非表示の切り替え(トグル)は追従しない: 統合先は常時表示になり、AAO がビルド時にソースの m_Enabled / m_IsActive アニメを無効化するため、統合したメッシュの ON/OFF トグルは効かなくなる(その衣装・装飾は常時表示になる)。切り替えを残したいメッシュはプレビューで「このメッシュは統合しない」を選ぶ。常時表示でよいものは「トグル固定」で表示固定にしてから統合するとよい
   - プレビュー表とスロット: レンダラーごとに「統合する/しない+理由」と統合後の想定 SkinnedMesh 数(例 5→2)を表示する。マテリアルスロットは統合時に同一マテリアルのサブメッシュがビルド時に重複排除される(アトラスと併用でさらに削減)
   - 反映のタイミング: 統合は NDMF ビルド時(Play/アップロード)に AAO が実行する。保存された _Opt プレファブ自体は統合前のメッシュ・スロット数のまま表示される。最終的な数値は SDK のビルド結果で確認すること
   - NDMFコンソールのAAO警告について: 変換直後の即時実測(1.4.0 以降)や ▶️(Play)・ビルド時に、NDMF コンソールへ AAO 発の警告(統合先に Root Bone / Anchor Override が設定されていない、統合ソース間の同名 BlendShape の値不一致)が出ることがある。統合先メッシュ(RARA_MergedMesh)まわりのもので、これらは 1.4.0 以前からビルド時に発生していたもの。即時実測で毎回コンソールを通るようになって見えやすくなっただけで、新しい不具合ではない。1.4.1 では統合先へ Root Bone と Anchor Override を自動設定して食い違いを抑え、アニメ未使用の同名 BlendShape の値衝突は現在値でメッシュへ焼き込んで自動固定する(現在値のまま焼くので見た目は変わらない)。アニメで動かしている BlendShape は表情などが壊れないよう自動固定せず、変換レポートで案内する。1.7.1 では、統合そのものが逆効果になるレンダラーを2種、自動で統合対象から外すようにした。(1) 多重描画(マテリアルスロット数>サブメッシュ数。例: 髪の mat_hair + FakeShadow)は、AAO がビルド時に多重描画を実体化(サブメッシュ複製)してポリゴンが増えるため統合しない。とくに PC はポリゴンの崖が 70,000 で共通(Good/Medium/Poor いずれも)で、超えると段差なく即 Very Poor になるため影響が大きい(実例では PC の _Opt 複製がビルド時に 68,674→78,948 三角形へ増え、70,000 を越えて Very Poor 化していた。AAO 警告「複数マテリアルによるマルチパスレンダリング…ポリゴン数が増えるなどの影響」)。(2) 統合対象間で material.(マテリアルプロパティ)アニメーションの対象が食い違うメッシュは、統合すると AAO がそのアニメを統合先全体へ波及させるため統合しない(AAO 警告「_UseMain2ndTex, _UseEmission へのアニメーションが異なります…意図していない部分にアニメーションが適用されてしまう可能性」。全ソースが同じ material. アニメ集合なら波及しないためそのまま統合する)。いずれも統合プレビューに理由を表示する。また、上記の同名 BlendShape の値不一致で「アニメーションで動かしているシェイプ」(例: ジャケットのトグルで動く shrink_for_jacket)は、実行時にアニメーションが値を決めるため統合対象間で現在値が食い違っていても多くの場合は無害で、変換レポートの案内にもその旨を明記した(見た目が意図と異なるときだけ該当メッシュを統合から除外する)。なお AAO が出す最適化結果のログ(「!」アイコン付きで削減量などを報告するもの)はエラーではなく正常な成果報告なので、赤字のエラーでなければ変換は成功している
7. 非破壊での生成(_Opt 複製 + プレファブ)
   - 元アバターを変更せず、「アバター名_Opt」のシーン複製を生成する。同時に同じ内容のプレファブアセットを保存する
   - 生成物は Assets/RARA/PCOptimizer/Generated/アバター名/ に保存される(Materials / Textures / Meshes)。再生成すると同じファイル(同じ GUID)へ内容だけ上書きされ、参照は壊れない
8. Quest対応コンバーターへの引き継ぎ
   - PC 軽量化の結果(_Opt)を、ワンクリックで姉妹ツール Quest対応コンバーターの変換フローへ渡せる

## PC(Windows)ランク閾値

VRChat SDK が PC(Windows / Standalone)アバターのランク判定に使う上限値。数値は SDK の判定アセットから直接読んだ実値。
各数値はそのランクの上限で、値がその数値以下なら少なくともそのランク以内に収まる。Poor の数値を超えると Very Poor になる。

| 項目 | Excellent | Good | Medium | Poor |
|------|-----------|------|--------|------|
| ポリゴン(三角形) | 32,000 | 70,000 | 70,000 | 70,000 |
| スキンメッシュ数 | 1 | 2 | 8 | 16 |
| メッシュレンダラー数 | 4 | 8 | 16 | 24 |
| マテリアルスロット数 | 4 | 8 | 16 | 32 |
| テクスチャメモリ(MB) | 40 | 75 | 110 | 150 |
| ボーン数 | 75 | 150 | 256 | 400 |
| PhysBone コンポーネント | 4 | 8 | 16 | 32 |
| PhysBone 対象トランスフォーム | 16 | 64 | 128 | 256 |
| PhysBone コライダー | 4 | 8 | 16 | 32 |
| PhysBone 衝突判定回数 | 32 | 128 | 256 | 512 |
| コンタクト(ローカル除く) | 8 | 16 | 24 | 32 |
| コンストレイント | 100 | 250 | 300 | 350 |

その他のカテゴリ(参考): アニメーター数 1 / 4 / 16 / 32、ライト 0 / 0 / 0 / 1(動的ライトが1つでもあると PC では最良でも Poor)、オーディオソース 1 / 4 / 8 / 8、コンストレイント深度 20 / 50 / 80 / 100。

ポリゴンについての注意: Excellent だけが 32,000 で、Good・Medium・Poor はすべて 70,000 と同じ。つまりポリゴン数だけで Medium や Poor になることはなく、32,001〜70,000 は Good、70,000 を超えると一気に Very Poor になる。PC で Very Poor はアップロード自体は可能だが、相手のパフォーマンス設定によっては非表示になる。

Quest / スマホ基準はこれよりずっと厳しい(例: Poor でもポリゴン 20,000、テクスチャメモリ 40MB、マテリアルスロット 4)。Quest 対応が必要なら姉妹ツール Quest対応コンバーターを使うこと。

## できないこと

- ポリゴン削減はしない。ポリゴンを減らすのは外部の減面ツールの役目。本ツールを使う前に 70,000 以下(Excellent 狙いなら 32,000 以下)にしておく
- シェーダー変換はしない。PC では元のシェーダー(lilToon 等)のまま使う。Quest / スマホ用の Mobile シェーダーへの変換が必要なら姉妹ツール Quest対応コンバーターへ
- ボーン数そのものを削る機能は持たない(AAO の Trace and Optimize が未使用ボーンの整理をビルド時に一部担当する)

## スロットが減らないとき

マテリアルスロット(マテリアル統合)が思ったより減らないときは、次のどれかに当てはまることが多い。原因はウィンドウの統合プレビューに理由として表示されるので、まずそこを確認するとよい。

- シェーダー系統が違う: lilToon と lilToonFur(ファー)/ lilToonTransparent(透過)/ カットアウトなど、実体の異なるシェーダー同士は統合できない。lilToon はインスペクタ上は1つに見えても、内部的にはアウトライン・ファー・透過などで別々のシェーダーになっており、系統が違うと同じ1枚にまとめられない
- カリングが違う: 片面(Cull Back)と両面(Cull Off)が混在していると別グループになる。「カリング差を無視」をオンにすると統合できる(統合後は両面描画になる)
- アウトライン有無が違う: アウトライン付きと無しは別シェーダーのため分かれる。「アウトライン」を「アウトラインを外して統合(推奨)」にすると統合できる(統合マテリアルはプレーンlilToonに揃い、服の輪郭線は消える)。「アウトライン付きに統一」でも統合できる(輪郭の無かった部分に付く。瞳・顔は自動回避)
- アニメで差し替えられる: アニメーションでマテリアルが切り替わるものは、実行時の差し替えと食い違わないよう統合対象外になる
- タイリングを使っている: UV が 0..1 範囲外、またはメインテクスチャにタイリング/オフセット設定があるマテリアルは統合対象外になる
- MeshRenderer が分散している: 別々のレンダラーに分かれたスロットの結合はビルド時に AAO(Trace and Optimize)が担当する。AAO が無い、または結合対象外のレンダラーは、統合してもスロットが残る
- SkinnedMesh が多くて減らない: レンダラーごとに 1 スロットずつ残る(1レンダラー×1マテリアル=1スロット)ため、SkinnedMesh 数そのものを減らさないとスロットも下がらない。このときは「メッシュ・トグル整理」の SkinnedMesh統合(顔以外を1つへ)で顔以外を1レンダラーへまとめると、SkinnedMesh 数もスロット数も同時に下がる(メッシュ数削減とスロット削減は同じ操作)

## 注意

- 診断ランクはエディタ上の状態で計算されるため、AAO 等がビルド時に統合・削減する分は反映されず、実際より悪めに出ることがある。最終確認は SDK のビルド結果で行うこと
- 瞳のテクスチャが消える場合: MA(Modular Avatar)の Material Setter 等で差し替えられる瞳マテリアルは、アトラス除外が自動で効く(最新版)。それでも消える場合は、そのマテリアルを手動でアトラス除外に指定する。アウトラインの黒縁が瞳・顔に付くのが原因なら、「アウトライン」を「アウトラインを外して統合」または既定の「アウトライン統合しない」にする(「アウトライン付きに統一」でも瞳・顔系は自動回避される)
- 色だけ材質の色ベイクとエミッションマスクは近似。マテリアルの色/発光を単色セルとしてアトラスに焼くため、細かなグラデーションや特殊な発色は正確には再現されない
- アトラス統合時はマットキャップ(lilToon の映り込み)が外れる。UV 再配置でマットキャップのマスクが壊れて全面が白飛びするため、統合するマテリアルからはマットキャップを外す。映り込みを残したいマテリアルは「アトラス除外」に指定する(統合されず、マットキャップも保たれる)
- アトラス統合はメッシュの UV を再マップしてスロットをまとめるため、タイリング(UV が 0..1 範囲外)を使うマテリアルや、アニメーションでマテリアルを差し替えるものは対象外になる
- テクスチャ縮小は解像度が落ちるため、寄与の小さい小物・マスク類から下げると画質への影響を抑えやすい
- 生成される _Opt はシーン複製とプレファブの両方。アップロードするのは _Opt の方(元アバターではない)
- 生成し直すと、シーンに残った古い複製(_Opt)は自動削除される。見た目が崩れた古い複製が残っている場合は削除して再生成する
- 設定を変えて再生成するとメッシュ・アトラスは新しいアセットになり、古い複製やプレファブ参照はそのまま以前の見た目で残る(参照を新しい生成物へ差し替えるには _Opt プレファブを使う)
