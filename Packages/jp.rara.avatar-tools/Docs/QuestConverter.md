# RARA Quest Converter

移行のお知らせ

操作は統合ウィンドウへ移行しました。メニューの RARA > アバター軽量化・Quest・iOS対応ツール を選ぶと統合ウィンドウが開き、上部のチップで対象を Quest(または PC・両方)に切り替えられます(変換エンジンは同じものを共有しています)。この単体UI(QuestConverterWindow)はメニューからは外しましたが、クラスと Open() API はコード互換のために残しています。

VRChat の PC 用アバターを Quest / スマホ(Android・iOS)対応に変換する Unity Editor 拡張。

本ツールは Quest / スマホ対応への変換専用。シェーダーを Mobile 系へ変換し、テクスチャに質感をベイクして Android / iOS で使える形にする。PC のまま性能ランクを上げたい(Very Poor を避けたい、Good を狙いたい)場合は、変換ではなく PC 基準のランク改善を行う姉妹ツール RARA PC軽量化ツール を使うこと。両ツールは共通エンジンを持ち、PC 軽量化の結果をそのまま本ツールの Quest 変換へ引き継げる。

主な特徴

- 元のアバターは一切変更しない非破壊変換。「アバター名_Quest」の複製を生成する
- lilToon(色調補正・影・エミッション等)や NonToon(ベース質感・影ランプ)、Poiyomi(メイン色調補正・ノーマル・エミッション。ロック済みも可)、Unity Standard/Unlit・VRM MToon(VRoid)の質感をテクスチャにベイクして VRChat/Mobile/Toon Standard へ変換(lilToon が最も忠実、他はプロパティ読み取り+ティント焼き込みの近似)
- 診断 → マテリアル個別設定 → 変換 → 結果チェックまでウィンドウひとつで完結
- メニューで切り替わる衣装・目・表情差分(Modular Avatar / Avatar Menu Creator 経由)も追従
- 複数マテリアルを1枚に統合するアトラス機能で、スロット数超過の Very Poor 回避を支援(実験的)
- ポリゴン削減は Meshia Mesh Simplification(Ram.Type-0 氏 / MIT)への連携方式。目標三角形数を指定してワンボタンで「_Quest」複製へ Meshia を追加し、ビルド時に高品質な簡略化を適用する(1.6.0 で自前の簡略化 UI は廃止)。Meshia 未導入時はパネルに導入手順を表示する

## 導入条件

- Unity 2022.3 系(VRChat 公式指定バージョン。2022.3.22f1 で動作確認)
- VRChat Avatars SDK 3.8.1 以降(VCC / ALCOM で作成したアバタープロジェクト。3.10.4 で動作確認)
  - 3.8.1 以降が必要なのは、変換先の VRChat/Mobile/Toon Standard シェーダーがこのバージョンで追加されたため
- lilToon(変換元アバターのマテリアルが lilToon の場合のみ必要。ベイクに lilToon 付属のベイク用シェーダーを使用する。2.3.4 で動作確認)
- レンダーパイプラインは Built-in のみ対応
- AvatarOptimizer(AAO)の事前導入を強く推奨(1.9 系で動作確認)
  - ツール自体は AAO なしでも動作するが、メッシュ結合はビルド時に AAO が担当する分業設計。素体+衣装の構成はスキンメッシュ数が Android の上限(2)を超えやすく、AAO の Trace and Optimize がないと Very Poor 判定になりやすい(既定でフォールバック表示・揺れ物停止。アップロード可否自体はサイズ上限のみで決まる)。実質必須と考えてよい
  - 導入は VCC/ALCOM にリポジトリを追加して行う。ブラウザで https://vpm.anatawa12.com/add-repo を開き、VCC/ALCOM が開いたら Add Repository → プロジェクトに AvatarOptimizer を追加 → Unity へ戻る
- Modular Avatar / NDMF は必須ではない(入っていれば MergeAnimator や Material Setter も変換対象として扱う)

## 導入方法

VCC / ALCOM から VPM パッケージ(jp.rara.avatar-tools)として導入する。詳しい手順は
リポジトリ直下の README.md「導入(VCC / ALCOM)」を参照。パッケージは
Packages/jp.rara.avatar-tools/ 以下に配置され、コンパイルが終わるとメニューバーに

追加の設定は不要。アンインストールは VCC / ALCOM の Manage Project からパッケージを外すだけでよい。

## 使い方

ウィンドウは上から順に進めば完了する構成になっている(対象選択 → プリセット → ① 診断 → ② 構成整理 → ③ PhysBone → ④ テクスチャ → ⑤ ポリゴン(Meshia連携) → ⑥ マテリアル → ⑦ 実行 → ⑧ ヘルプ)。対象選択の後は、目的に近いプリセット(おまかせQuest対応 / PCを最低Poorまで軽量化(Very Poor回避) / PCをGoodまで軽量化 / フル両対応(PC+Quest))を押すと、対象と目標ランクがまとめて設定される(あとから各項目で微調整できる)。

1. シーンに置いたアバター(VRCAvatarDescriptor 付き)をウィンドウで選択する(選択すると自動で診断が走る)
2. 診断結果でランク・アップロード可否・推定ダウンロードサイズを確認する
3. 透過は既定で自動再現される(乗算/加算で近似)。特定のマテリアルだけ扱いを変えたいときは、マテリアル設定の「透過マテリアルの既定処理」や各マテリアルの個別指定で調整する
4. サイズが 10MB を超えそうなら、削減提案に従ってテクスチャ縮小や Quest除外を設定する
5. スロット数が超過するアバターはアトラス統合を有効にする
6. ポリゴン数が多い場合はポリゴン削減(Meshia連携)で目標三角形数(目安 7,500 / 10,000 / 15,000 / 20,000)を指定し、ワンボタンで「_Quest」複製へ Meshia を追加する。削減はビルド時に Meshia が適用するため、結果は即時実測/実測レポートで確認する。Meshia 未導入時はパネルに導入手順が出る。先にメッシュ削減(AAO連携)・衣装・トグル整理で不要なメッシュを消してから使うと画質を保ちやすい
7. 実行セクションの「この実行で行われること」サマリーを確認して「実行(選択した対象へ生成)」を実行、レポートと「変換結果チェック」(変換前後のサムネイル比較)を確認する
8. アップロード時は SDK コントロールパネルでビルドターゲットを Android に切り替え、アップロード手順ガイドの「_Questだけをアクティブ化」で「_Quest」だけをアクティブにしてから、その「_Quest」の方をアップロードする。PC 版と同じ Blueprint ID を使うと、1 つのアバターで PC / Quest 両対応になる。SDK の「Multi-Platform Build & Publish」(複数プラットフォーム同時ビルド)は使わない(本ツールは PC 版と Quest 版を別々の複製でアップロードする方式のため、プラットフォームは1つずつ選ぶ)

## できること

1. Quest適合診断
   - VRChat SDK の性能統計 API を Android/iOS 基準で実行し、総合ランクとカテゴリ別ランクを表示する
   - 非 Mobile シェーダーのマテリアル、非対応コンポーネント、Android 縮小設定のないテクスチャを一覧表示する
   - EditorOnly タグ付きのオブジェクトと Quest除外に登録したオブジェクトは、実際のビルドと同様に除外した状態で計算する
2. サイズ診断(推定)
   - 推定ダウンロードサイズを、独立した2つのアップロード上限「圧縮後(ビルド後ダウンロード)10MB」と「展開後(非圧縮メモリ)40MB」の両方と比較して表示する(テクスチャ / メッシュ / アニメーションの内訳付き。あくまで目安)。圧縮後が収まっていても展開後が40MBを超えるとSDKがアップロードをブロックするため、両方を並べて超過している側を示す
   - メインテクスチャだけでなく、エミッションマップ・ノーマルマップ・マスク類も含めた全テクスチャが対象。寄与の大きいものを上位から一覧表示し、削減提案を削減量順に表示する
   - テクスチャの提案は「適用」(個別)/「提案をすべて適用」(一括)で「縮小計画」に登録できる
   - 「上限内まで自動調整(圧縮後10MB・展開後40MB)」ボタンで、圧縮後・展開後の両上限を同時に狙い、寄与の大きいテクスチャから順に解像度を段階的に下げる計画を自動作成し、確認ダイアログで内容(どれを何pxにするか)を見てから一括登録できる。顔・体系の名前を持つテクスチャは後回しにして画質を優先する。テクスチャを最小(縮小フロア)まで下げても上限に届かない場合(展開後がメッシュ・ブレンドシェイプ支配のときなど)は、その旨を正直に表示し、テクスチャ縮小だけでは収まらないこと(Quest除外・メッシュ削減・ブレンドシェイプ整理の併用が必要)を案内する
   - 縮小計画に登録されたテクスチャは、変換時に低解像度のコピーが Generated に生成され、変換後マテリアルがそれを参照する。元のテクスチャとそのインポート設定には一切触れない。「縮小計画をクリア」でいつでも取り消せる
3. マテリアル別の変換方法指定
   - アバターの全マテリアルが一覧表示される(透過/カットアウト/パーティクル/アニメ使用/メニュー・ギミック参照などのバッジ付き)
   - マテリアルごとに変換方法を選べる: 自動(推奨) / Toon Standard / Toon Lit / MatCap Lit(金属・ツヤ向け) / パーティクル加算 / パーティクル乗算 / 非表示 / 変換しない
   - 各行に「変換後にどうなるか」の説明が表示されるので、実行前に結果を把握できる
   - Renderer 以外のコンポーネント(Modular Avatar の Material Setter、Avatar Menu Creator 等)から参照されているマテリアルも収集・変換・差し替えされる(メニューで切り替わる目・衣装・表情差分の消失を防ぐ)
4. 半透明マテリアルの自動再現(既定)
   - アルファブレンド透過マテリアル(表情用チーク、涙、ガラス等)は、Quest 版では乗算/加算のパーティクルシェーダーで半透明を近似再現する。設定不要で、チーク・涙・ガラス系がそれらしく表示される(旧バージョンの「非表示」から既定を変更)
   - 暗めの透過(チーク・頬染め・影・すりガラス等)は VRChat/Mobile/Particles/Multiply(乗算)で、明るい透過(涙・アイハイライト・光り物等)は VRChat/Mobile/Particles/Additive(加算)で近似する。メインテクスチャの明るさから自動で判定する
   - 既定処理は「3. マテリアル設定」の「透過マテリアルの既定処理」で切り替えられる: 自動で半透明を再現(推奨) / 非表示にする(最軽量) / 不透明に変換。マテリアル単位の上書きも同じ一覧で行える
   - 近似の限界: 乗算は背景を暗くする方向のみ、加算は明るくする方向のみで、色味は保持されるが(無彩色にはならない)明暗のどちらかに寄る。加算は暗いワールドで発光して見える。ちょうど中間の半透明(白いすりガラス等)は背景次第で見え方が変わる
   - 大型メッシュと髪(ポリゴン数が多い、または名前に hair/髪 等を含む)は保護対象で、既定処理に関わらず常に不透明として変換される(透過髪が消える・板化する事故を防ぐ。近似も非表示もしない)
   - 顔は自動保護される(1.4.2から)。顔の基底マテリアルは、透過設定でも半透明の自動再現(パーティクル化)や非表示の対象にせず、既定処理に関わらず常に不透明(Toon Standard)として変換される(顔が白飛び・消失する事故を防ぐ)。顔はアバターディスクリプタのビセーム/まばたきメッシュ、または名前に face・顔 等を含むメッシュ/マテリアルで判定する。チーク・涙・ハイライトなどのオプション重ねマテリアル(名前で判定)は従来どおり乗算/加算で近似する。既存の大型メッシュ・髪の保護と同じ扱い
   - カットアウト(髪の透け先端など)は形状が意味を持つため対象外で、不透明として変換される
   - 「非表示にする」を選んだ場合は、オブジェクト全体が透過なら EditorOnly 化(描画コストごと削減)、一部スロットだけ透過なら実質不可視のマテリアルに差し替えて隠す
5. 衣装・トグル整理(メッシュ/スロット削減、Quest・PC両対応)
   - トグルで切り替える衣装・アクセサリを検出して一覧表示する。各グループを「維持 / 常時表示 / 非表示除去」から選べる
   - 「常時表示」= 常に ON 固定にしてトグルアニメを外す。AAO がビルド時に結合できるようになり、スキンメッシュ数・マテリアルスロット数を大きく削減できる(そのトグルは無くなる)
   - 「非表示除去」= そのメッシュを削除(EditorOnly)。「現在の表示状態で固定(全トグル)」ワンクリックボタンあり
   - 各グループには「非表示固定で減る専有アセット」の削減量チップ(圧縮後 -◯MB / 展開後 -◯MB)を表示する。専有アセット=そのグループのサブツリー内のレンダラーだけが使い、サブツリー外では一切使われないテクスチャ・メッシュ。共有アセットしか持たないグループは「単独では減らない(共有テクスチャのみ)」と表示され、そのグループ1つを非表示にしてもサイズは減らない(同じテクスチャを使う他のメッシュが残るため)。一覧を専有削減量(展開後)の大きい順に並べ替えることもできる
   - サイズが上限を超えているときは、削除候補を専有削減量の大きい順に提示するガイドを表示する。ガイドの「上位◯件でカバーできる見込み」の合計は、親子(入れ子)の衣装グループを二重計上しない(親を非表示にすれば子のサブツリーも一緒に消えるため、重複するグループはカバー合計に含めない)
   - Quest 対応の仕組み上、衣装トグルを分離メッシュのまま持つとメッシュ2・スロット4の上限に入らない。ここで固定すれば上限内に収められる
   - PC のままランクを上げたい場合は、姉妹ツール RARA PC軽量化ツール を使うこと。シェーダー/テクスチャ変換をせずに、トグル整理・PhysBone マージに加えてテクスチャメモリ削減やマテリアル統合まで行い、PC(Windows)基準でランクを改善する(出力は _Opt 複製とプレファブ)。従来この機能をこのツールの「PC最適化のみ」モードとして提供していたが、PC 向けの機能はそちらへ集約した
   - 安全設計: 元アバターの共有アニメクリップは変更しない(クローン側のクリップだけ編集)。m_IsActive 以外のアニメ(ブレンドシェイプ等)は維持される
6. アトラス統合(実験的)
   - 互換性のある複数の変換済みマテリアルを 1 枚のアトラステクスチャ+1 マテリアルに統合し、メッシュの UV を再マップする
   - マテリアルスロット数と描画コストが大きく減るため、スロット数超過で Very Poor になっていたアバターを Medium 圏内に入れられる可能性がある
   - メッシュ結合そのものは AAO がビルド時に行う。本機能でスロットが同一マテリアルに揃うため、AAO の統合効果が最大化される
   - アニメーションでマテリアル差し替えされるもの、UV が 0..1 範囲外(タイリング使用)のものは自動で対象外になる(理由付きで表示)
   - 既にQuest対応(VRChat/Mobile/Toon Standard・Toon Lit)のマテリアルもアトラス統合の対象になった(1.8.0)。購入したQuest対応済みアバターや、元アバターを持たない過去の変換結果(RARA_Atlas_* を含む)でも、そのまま使いつつスロットを削減できる。MatCap Lit・パーティクル系は従来どおり対象外。過去のアトラス出力が含まれる場合は「元アバターから変換し直すとより効率的に統合できます」と案内する
7. メッシュ削減(AAO連携)
   - 服の下に隠れて見えない肌などのポリゴンを削減する。アバターが持つ「shrink(縮小)ブレンドシェイプ」を自動検出し、AAO の RemoveMeshByBlendShape を複製に自動セットアップできる(見えない部分だけをビルド時に削除)
   - 検出の仕組み: ① ブレンドシェイプ名が隠し/縮小系トークン(shrink / hide / 縮小 / 非表示 / 削除 / 消し / 隠し 等)を含むもの、または部位トークン(肌 / 素体 / パンツ / インナー 等)とオフ系トークン(off / オフ / なし 等)が同居するものを候補にする。② 顔まわり(顔 / 目 / 眉 / 口 / 涙 / まつ毛 / ほくろ、あ・い・う・え・お、まばたき 等)は名前・ジオメトリの両面で候補から除外し、表情・顔ディテール・MMD 標準モーフ・リップシンクの誤削除を防ぐ。③ 名前が一致しなくても、最終フレームで頂点の大部分(全体の約40%以上)を強く動かす形状だけを幾何判定で候補にする(保守的な閾値。頂点数が少なすぎるメッシュは対象外)
   - 穴に注意: 削除されるのは選んだシェイプが動かす領域だけ(=制作者が「隠れる前提」で設計した箇所)。そのため、その服を後から非表示にすると、下にあった肌が無くなって穴が開いて見える。削除は変換後の複製に付く AAO の Remove Mesh By BlendShape がビルド時に適用するため、そのコンポーネントのプレビューで削除後の状態を確認できる(元アバターは無改変)
   - Trace and Optimize が無ければ複製に追加してビルド時最適化を有効化する(付与のオン/オフは⑦実行の実行オプション「AAO(Trace and Optimize)を複製へ付与する」の1か所にまとまっている)
   - shrink ブレンドシェイプが無いアバター向けに、AAO の Remove Mesh in Box(箱で範囲指定削除)/ By Mask(マスク画像で削除)の手引きを表示する
   - 顔の表情・MMD・リップシンク用ブレンドシェイプは誤検出しない安全策付き
   - SkinnedMesh統合(顔以外を1つへ / グループ指定): 顔(ビセーム/まばたき)以外の SkinnedMeshRenderer を AAO の Merge Skinned Mesh で 1 つへ統合し、SkinnedMesh 数とマテリアルスロット数を確実に減らす(Quest Poor 上限 SkinnedMesh 2 / スロット 4 対策)。「統合しない / 顔以外を統合(推奨) / グループ指定で統合」から選ぶ。既定は統合(旧設定から引き継いだ場合は統合しない=従来動作のまま)
   - グループ指定モード: 顔以外を一律に1つへまとめるのではなく、レンダラーごとにグループ1〜8を割り当て、同じグループのレンダラーだけを1つのメッシュへ統合する。素体と衣装を別メッシュに保ちたい、切り替えたい衣装だけ分けておきたい、といった場合に使う。顔メッシュは常に自動保護されてグループへ割り当てできず、どのグループにも割り当てなかったレンダラーは統合されない。プレビューにグループごとの「n枚 → 1枚」と、統合後の内訳(グループ統合 + 顔(自動保護) + 統合しない = 統合後の概算枚数)を表示する
   - 統合するもの: 顔以外の通常の SkinnedMeshRenderer(素体・衣装・髪・アクセサリ等)。統合しないもの: 顔メッシュ(ディスクリプタの VisemeSkinnedMesh / まばたきメッシュ=分離維持)、Cloth が付いたメッシュ(布シミュレーションを保持)、EditorOnly(ビルド除外)、プレビューで個別に「統合しない」を選んだメッシュ、多重描画(マテリアルスロット数>サブメッシュ数。統合するとビルド時にポリゴンが増えるため自動除外。1.7.1〜。ただし 1.9.0 で上描きスロットの削除により統合可能になった=下記)、統合対象間で material.(マテリアルプロパティ)アニメーションの対象が食い違うメッシュ(統合すると統合先全体へ波及するため自動除外。1.7.1〜)
   - 上描きスロットの削除で統合(1.9.0): 1.7.1 で一律に統合対象外にしていた多重描画(スロット数>サブメッシュ数)のレンダラーを、上描き(余剰)スロットの削除で統合できるようにした。① 何も描かない上描きは自動削除して統合 — 余剰スロットのうち、スロットが空、または Quest 対応で非表示化される疑似影 FakeShadow / アウトラインのみなど何も描かないものは、統合の直前に複製から自動で削除する。多重描画でなくなるため、ポリゴンを増やさず・見た目も変えずに通常どおり統合される(Quest の疑似影付き髪などが操作なし・見た目の変化なしで自動的に統合され、統合後の概算が 3 に留まっていたものが 2 になる)。② 見える上描き(前髪影など)はチェックで削除して統合 — 実際に重ねて描かれている余剰スロット(例: PC の FakeShadow)は、統合プレビュー各行の「上描きスロットを削除して統合(前髪影などの重ね描き効果が消えます)」チェック(既定オフ)で削除して統合できる。重ね描き効果は失われるが、SkinnedMesh 数・スロット数を確実に減らせる(チェックすると理由が「上描きスロット削除により統合します」に変わる)。③ 差し替えアニメ対象は不可 — マテリアル差し替えアニメ(m_Materials.Array.data[N])の対象レンダラーは、削除するとスロット番号がずれてアニメが壊れるため、自動削除・チェックのいずれの対象にもしない(「マテリアル差し替えアニメがあるためスロット削除不可」と表示)。削除するのは常に余剰スロット(サブメッシュ数を超える分)だけで、基底スロットには手を付けない。複製のみで元アバターは変更しない
   - マテリアルアニメーションを無効化して統合(1.10.0): 1.7.1 の波及ガードで「マテリアルアニメーションが統合先全体へ波及するため統合しません」として外されたメッシュ(エミッション切り替え等の material.(マテリアルプロパティ)アニメを持つシャツ・ネクタイなど)を、そのアニメを無効化することで統合できるようにした。ガードで除外された各行に「マテリアルアニメーションを無効化して統合(この切り替え演出は動かなくなります)」チェック(既定オフ)を表示し、オンにすると、そのレンダラーに向いた material.* アニメを複製側のアニメーションクリップからのみ無効化(除去)して統合する。無効化により波及の懸念がなくなるため統合先へまとまり、SkinnedMesh 数が減る(例: 顔以外がシャツ・ネクタイのエミッション演出で分かれて統合後 4 になっていたものを、両方チェックして Poor 上限 2 に収められる)。チェックした行は理由が「マテリアルアニメーションを無効化して統合します」に変わり、統合後の概算へ反映される。無効化は複製に対してのみ行い、元アバターや共有するクリップ・コントローラーは一切書き換えない(衣装・トグル整理の m_IsActive 無効化と同じクローン専用の複製・差し替えの仕組みを共用)。チェックした切り替え演出は動かなくなる(固定表示になる)。変換レポートに「'X': マテリアルアニメーションを無効化して統合しました(無効化: _UseEmission ほか n件 / クリップ: A, B)。該当の切り替え演出は動かなくなります。」を出す。マテリアル差し替えアニメ(m_Materials.Array.data[N])は従来どおり対象外(チェックは出ない)で、無効化できるのは material.(プロパティ値)のアニメだけ
   - 表情は保たれる: 顔メッシュを分離維持するため、ビセーム・まばたき・表情ブレンドシェイプはそのまま動く。顔以外に残るブレンドシェイプ(靴下オフ等)・マテリアルアニメも、統合時に AAO がブレンドシェイプ名の改名・アニメの再パスを行うため追従して動き続ける(同名衝突は別名化=RenameToAvoidConflict で回避)
   - 再処理時の残骸削除(1.8.1): 変換済みアバターを再処理する場合、前回の統合先の残骸(命名が RARA_MergedMesh / RARA_MergedMesh_G{n} で、メッシュ未設定のもの)は変換時に複製から自動で削除される(古い AAO Merge Skinned Mesh ごと消えるため二重統合を防ぐ。元アバターは変更しない)。統合プレビューには「前回の統合先(残骸)。実行時に複製から削除されます」と表示し、統合後の概算からも除外する。ユーザーが自作したメッシュ未設定レンダラー(命名不一致)は削除しない
   - ただし表示/非表示の切り替え(トグル)は追従しない: 統合先は常時表示になり、AAO がビルド時にソースの m_Enabled / m_IsActive アニメを無効化するため、統合したメッシュの ON/OFF トグルは効かなくなる(その衣装・装飾は常時表示になる)。切り替えを残したいメッシュはプレビューで「このメッシュは統合しない」を選ぶ。常時表示でよいものは「衣装・トグル整理」で表示固定にしてから統合するとよい
   - プレビュー表: レンダラーごとに「統合する/しない+理由」と統合後の想定 SkinnedMesh 数(例 5→2)を表示する。マテリアルスロットは統合時に同一マテリアルのサブメッシュがビルド時に重複排除される(アトラスと併用でさらに削減)
   - 反映のタイミング: この統合は NDMF ビルド時(Play/アップロード)に AAO が実行する。保存された複製プレファブ自体は統合前のメッシュ・スロット数のまま表示される(既存のブレンドシェイプ削除・自動統合と同じ挙動)。最終的な数値は SDK のビルド結果で確認すること
   - NDMFコンソールのAAO警告について: 変換直後の即時実測(1.4.0 以降)や ▶️(Play)・ビルド時に、NDMF コンソールへ AAO 発の警告(統合先に Root Bone / Anchor Override が設定されていない、統合ソース間の同名 BlendShape の値不一致)が出ることがある。これらは 1.4.0 以前からビルド時に発生していたもので、即時実測で毎回コンソールを通るようになって見えやすくなっただけで、新しい不具合ではない。1.4.1 では統合先へ Root Bone と Anchor Override を自動設定して食い違いを抑え、アニメ未使用の同名 BlendShape の値衝突は現在値でメッシュへ焼き込んで自動固定する(現在値のまま焼くので見た目は変わらない)。アニメで動かしている BlendShape は表情などが壊れないよう自動固定せず、変換レポートで案内する。1.7.1 では、統合そのものが逆効果になるレンダラーを2種、自動で統合対象から外すようにした。(1) 多重描画(マテリアルスロット数>サブメッシュ数。例: 髪の mat_hair + FakeShadow)は、AAO がビルド時に多重描画を実体化(サブメッシュ複製)してポリゴンが増えるため統合しない(AAO 警告「複数マテリアルによるマルチパスレンダリング…ポリゴン数が増えるなどの影響」。実例では PC 複製が 68,674→78,948 三角形へ増え 70,000 の崖を越えていた)。(2) 統合対象間で material.(マテリアルプロパティ)アニメーションの対象が食い違うメッシュは、統合すると AAO がそのアニメを統合先全体へ波及させるため統合しない(AAO 警告「_UseMain2ndTex, _UseEmission へのアニメーションが異なります…意図していない部分にアニメーションが適用されてしまう可能性」。全ソースが同じ material. アニメ集合なら波及しないためそのまま統合する)。いずれも統合プレビューに理由を表示する。また、上記の同名 BlendShape の値不一致で「アニメーションで動かしているシェイプ」(例: ジャケットのトグルで動く shrink_for_jacket)は、実行時にアニメーションが値を決めるため統合対象間で現在値が食い違っていても多くの場合は無害で、変換レポートの案内にもその旨を明記した(見た目が意図と異なるときだけ該当メッシュを統合から除外する)。なお AAO が出す最適化結果のログ(「!」アイコン付きで削減量などを報告するもの)はエラーではなく正常な成果報告なので、赤字のエラーでなければ変換は成功している
8. ポリゴン削減(Meshia連携)
   - ポリゴン削減は Meshia Mesh Simplification(Ram.Type-0 氏 / MIT)へ連携する方式。本ツール自前の頂点間引き UI は 1.6.0 で廃止し、本ツールは目標三角形数の指定と「_Quest」複製への Meshia コンポーネント追加を担当する(実際の簡略化は Meshia がビルド時に行う)
   - 目標三角形数(目安 Excellent 7,500 / Good 10,000 / Medium 15,000 / Poor 20,000)を指定し、ワンボタン(「_Quest 複製へ Meshia を追加」)で複製へ Meshia のメッシュ簡略化コンポーネントをセットアップする。目標へ合わせた配分は Meshia 側の設定として書き込まれる
   - 反映のタイミング: 簡略化は NDMF ビルド時(Play / アップロード)に Meshia が実行する。エディタ上の診断値や保存された複製プレファブには反映されないため、結果は変換直後の即時実測(1.4.0 以降)または実測レポート(▶️ Play / SDK ビルド)で確認する
   - Meshia は頂点数・UV シーム・形状特徴を考慮した高品質な簡略化を行う。崩れやすい/見せ場の部位(顔・髪など)は、追加後に Meshia コンポーネント側で目標三角形数や保護設定をレンダラー単位で微調整できる(EditMode プレビューで品質を確認できる)
   - Meshia 未導入時: パネルに導入手順(VPM リポジトリの追加)を表示する。導入は VCC / ALCOM にリポジトリ https://ramtype0.github.io/VpmRepository/ を追加して行う(公式ドキュメント https://ramtype0.github.io/Meshia.MeshSimplification/ )
   - 移行注記: 旧バージョンで保存した自前のポリゴン削減計画(レンダラーごとの配分)は 1.6.0 では適用されない。ポリゴンを削りたい場合は、Meshia を導入して目標三角形数を指定し直す
   - 先にメッシュ削減(AAO連携)・衣装・トグル整理で不要なメッシュを消してから使うと、同じ目標でも削る量が減って画質を保ちやすい
   - 元アバターは変更されず、Meshia は「_Quest」複製にのみ追加される(Unity Cloth が付いたメッシュなど Meshia 側で扱えない対象は簡略化されない)
9. Quest除外オブジェクト(Quest版だけの軽量化)
   - 登録したオブジェクトは _Quest 複製でのみ EditorOnly 化され、ビルドから完全に除去される。PC 版には一切影響しない
   - 残すメッシュのボーンやアイトラッキング・リップシンクの参照を巻き込む場合は警告が出る
10. Quest対応版の生成(非破壊変換)
   - マテリアル変換: lilToon の色調補正・2nd/3rd レイヤーをベイクし、Toon Standard(既定: 影ランプ・ノーマル・エミッション・マットキャップ・最低明るさを引き継ぐ)または Toon Lit(最軽量)へ変換。エミッションの HDR 強度も2倍まで引き継ぐ(マットキャップの引き継ぎは下記「マットキャップ」参照)
   - NonToon(jp.lilxyzw)対応: lilToon と同様に自動変換する。ベーステクスチャ・ノーマルマップを引き継ぎ、影グラデーションを影ランプテクスチャへ生成、マットキャップは近似変換する。NonToon にはエミッションが無いためエミッションは付かない。アウトライン・ファーは Quest 非対応のため外れる(透過/カットアウトはレンダリングモードで判定するため、半透明の再現・非表示化も lilToon と同じ扱いになる)
   - Poiyomi(.poiyomi/Poiyomi Toon)対応: lilToon と同様に自動変換する。メインカラー(色調補正=色相/彩度/明度/ガンマ)をベイクし、ノーマルマップ・エミッションを引き継いで Toon Standard へ変換する。影ランプは SDK 既定を使い、マットキャップ・リム・アウトライン・シャドウ調整などの追加表現は再現されない(一部の質感は既定値の近似になる)。半透明/カットアウトはブレンド設定(_SrcBlend/_DstBlend・_ZWrite・_Cutoff 等)から判定するため、半透明の再現・非表示化も lilToon と同じ扱いになる。ロック済み(Lock In / シェーダー最適化済み)のマテリアルもそのまま変換できる。ロック時に剥がされたテクスチャはマテリアルタグから復元し、焼き込まれて失われたプロパティは既定値へ縮退する(この場合は一部の質感が既定値になる旨がレポートに出る)
   - Unity Standard / Unlit 対応: メインカラー・ノーマルマップ・エミッション・カリングを引き継いで Toon Standard / Toon Lit へ変換する。メタリック/スムースネスは無視するため、金属パーツはマテリアル設定で MatCap Lit を選ぶと映り込みを付けられる。透過/カットアウトはブレンド設定から判定するため、半透明の再現・非表示化も他ファミリーと同じ扱いになる(色調はメインテクスチャへ焼き込むメインテクスチャ基準の近似変換)
   - VRM MToon(VRoid)対応: メインカラー・ノーマルマップ・エミッション・カリングを引き継いで Toon Standard へ変換する。MToon の SphereAdd(スフィアマップ加算)は Toon Standard の内蔵マットキャップへ加算で引き継ぐ。カットアウト/透過方針は従来ポリシーに準拠する(VRoid 由来のアバターをそのまま Quest 対応にできる)
   - パーティクル対応: ParticleSystem / TrailRenderer / LineRenderer のマテリアルは不透明化せず Mobile/Particles 系へ自動変換。メッシュと兼用のマテリアルは 2 種類に変換して正しく割り当てる
   - アニメーション変換: マテリアル差し替えアニメ(FX 等)を検出し、コントローラーとクリップを複製して変換後マテリアルを参照させる
   - コンポーネント処理: Unity コンストレイント→VRCConstraint 変換、Cloth / Camera / Light / AudioSource / 物理 / DynamicBone / FinalIK 等の削除、設定が一致する兄弟 PhysBone チェーンのマージ(揺れを維持したままコンポーネント数を削減。既定オン。このマージは本ツール独自実装で AAO の Merge PhysBone は使用しない)、マージ後も Poor 上限を超える場合の超過分の自動削除(任意)
   - PhysBone 設定セクション: マージ適用後の構成(髪ルート配下 12本→1、スカート配下 8本→1、マージ対象外とその理由)を変換前にプレビュー表示する
     - Quest の Poor 上限は5項目ある: コンポーネント数 8 / 影響 Transform 数 64 / コライダー数 16 / 衝突判定回数 64 / コンタクト数 16。このどれか1つでも超えるとその分野が Very Poor になり、Quest 実機では PhysBone・コンタクト・コンストレイントがまとめて無効化される(揺れもの・Contact が全停止する)。コンポーネント数だけを見て 8 本に収めても、Transform やコライダー・コンタクトが上限を超えていれば同じく全停止するため、5項目すべてを同時に見て絞り込む必要がある
     - 既定はすべて残す方式(KeepAll): 各 PhysBone の「残す」チェックは既定でオン(チェック=残す)で、外したいものだけチェックを外す(=削除)。削除にしたものは別枠の一覧に移り、「戻す」で復元できる
     - 「優先度で自動選択」ボタン: Hair Root / Hair Front / hair side / Breast などの名前パターンで優先度を判定し、上位から順に残す。1.11.0 で、従来のコンポーネント数(8個)だけでなく、影響 Transform 数(64)・コライダー数(16)・衝突判定回数(64)・コンタクト数(16)の5制約すべてが Poor 上限に同時に収まるまで自動で絞り込むよう拡張した(どれか1つでも超えると実機で全停止するため、コンポーネント数だけを見ていた従来方式では取りこぼしがあった)
     - 未参照コライダーの自動削除(1.11.0): どの PhysBone からも参照されていない PhysBone コライダーは、コライダー数・コンポーネント数を消費するだけで揺れに寄与しないため、変換時に複製から自動で削除する(元アバターは無改変)
     - コンタクト(Contact)の整理(1.11.0): VRCContactSender / VRCContactReceiver も PhysBone と同様に「残す / 削除」リストで取捨選択でき、「優先度で自動選択」はコンタクトも Poor 上限(16)に収まるよう自動選定する(ローカル専用のコンタクトは集計対象外)
     - 5本のメーター表示(1.11.0): コンポーネント・影響 Transform・コライダー・衝突判定回数・コンタクトの5項目それぞれについて、選択後の値と Poor 上限を並べたメーターを表示する。上限を超えている項目は赤くなり、実機で全停止を招く分野が一目で分かる
     - 積極マージ(既定オン): 設定が完全一致しない兄弟チェーンも先頭の設定に統一してマージ(差異があるものは「設定差異あり」バッジ表示)。アニメ制御・パラメータ使用などの安全ゲートは維持
     - 選択後の個数が Medium 6 / Poor 8 に対してリアルタイム表示され、ランク別のコンポーネント上限ラダー(Excellent 0 / Good 4 / Medium 6 / Poor 8)と次ランクまであと何個削ればよいかの目安も併記される。上記5項目のどれか1つでも Poor 上限を超えると赤の強警告(モバイルは Very Poor で PhysBone・コンタクト・コンストレイントが全て無効化されるため)を出し、変換レポートでもエラー表示される
   - 変換できないもの: TextMeshPro は警告して Quest除外を提案。壊れたシェーダーはエラー報告してスキップ。いずれも変換全体は止めない
11. Android/iOS ビルド時の自動クリーンアップ(Auto Fix 相当)
   - FaceEmo 等がビルド時に注入する非対応コンポーネント(AudioSource 等)を、ビルド前処理で自動削除する。SDK の赤アラートで手が止まらない
   - ビルド用のコピーにのみ適用されるため、シーンのアバターは変更されない。PC ビルドでは何もしない
12. 生成物は Assets/RARA/QuestConverter/Generated/アバター名/ に保存される
   - 同じアバターを再変換すると、生成アセットは同じファイル(同じ GUID)へ内容だけ上書きされる。以前の _Quest 複製や、生成マテリアルを手動で割り当てた箇所の参照は壊れない
   - 例外はアニメーターコントローラーのみ(技術的制約で作り直しになる。旧 _Quest 複製のアニメ参照だけが切れるため、複製は新しいものに置き換えること)
   - 使われなくなった前回の生成物はレポートに一覧表示される(自動削除はしない)
   - 生成し直すと、シーンに残った古い複製(_Quest)は自動削除される。見た目が崩れた古い複製が残っている場合は削除して再生成する
   - 設定を変えて再生成するとメッシュ・アトラスは新しいアセットになり、古い複製やプレファブ参照はそのまま以前の見た目で残る(参照を新しい生成物へ差し替えるには _Quest プレファブを使う)

## 対応元シェーダー(変換元)

変換元(PC 側)のマテリアルが次のシェーダーファミリーなら、質感を読み取って Quest / スマホ対応の Toon Standard / Toon Lit へ自動変換する。lilToon が最も忠実に再現でき、それ以外のファミリーは「プロパティ読み取り+メインテクスチャへのティント焼き込み」による近似変換になる(色調補正は結果色をテクスチャへ焼き込んで近づけるため、PC 版と完全一致はしない)。1.3.0 で Unity Standard/Unlit・VRM MToon(VRoid)を変換元として追加し、Poiyomi Toon はロック済みマテリアルまで対応した。

- lilToon — 最も忠実。色調補正・影(2nd/3rd)・エミッション・マットキャップ・最低明るさ・カリングをベイクして引き継ぐ(付属のベイク用シェーダーを使用)
- NonToon(jp.lilxyzw) — ベース色・ノーマル・影ランプ・マットキャップ(乗算/加算の2枠→1枠に近似)・カリングを引き継ぐ。エミッションは非搭載のため付かない
- Poiyomi Toon(ロック済み含む) — メイン色(色相/彩度/明度/ガンマ)をティント焼き込みし、ノーマル・エミッション・カリングを引き継ぐ。影ランプは SDK 既定で、リムや独自の陰影は再現しない。ロック済み(最適化済み)も変換でき、焼き込みで失われたプロパティは既定値へ縮退する
- Unity Standard / Unlit — メイン色・ノーマル・エミッション・カットアウト/透過方針・カリングを引き継ぐ。メタリック/スムースネスは無視する(金属パーツはマテリアル設定で MatCap Lit を選ぶと映り込みを付けられる)
- VRM MToon(VRoid) — メイン色・ノーマル・エミッション・カットアウト/透過方針・カリングを引き継ぐ。SphereAdd(スフィアマップ加算)は Toon Standard の内蔵マットキャップへ加算で引き継ぐ

共通の方針:

- カットアウト/透過の扱いは従来の透過ポリシーに準拠する(半透明は乗算/加算で近似、大型メッシュ・髪は保護して不透明化、カットアウトは塗りつぶし)
- ノーマルマップ・エミッション・マットキャップ・カリングはファミリー横断で引き継ぐ(マットキャップは Toon Standard の内蔵マットキャップへ。詳細は下の「マットキャップ」参照)

近似・非対応(いずれも変換全体は止めず、変換レポートに警告を出す):

- シェーダー固有の色調補正・デカール・ラメ(グリッター)などの特殊表現は再現されない(結果色の近似へ丸めるか、警告して省く)
- Unity Standard のメタリック/スムースネスは無視される(金属感が要るパーツは MatCap Lit を推奨)
- アウトラインは従来どおり Quest 非対応のため外れる(警告)
- lilToon 以外は結果色をテクスチャへ焼き込む近似のため、ライティング応答や特殊効果は PC 版と一致しない

## マットキャップ(金属・ツヤの映り込み)

金属パーツやツヤのある面に使う「マットキャップ(MatCap)」を Quest / iOS でも近い見た目で再現する。lilToon のマットキャップは 1.1.0 から Toon Standard の内蔵マットキャップへ自動で引き継がれ、必要なパーツだけ専用の MatCap Lit シェーダーへ変換することもできる。

1. Toon Standard への自動引き継ぎ(既定)
   - lilToon のマットキャップは、既定で変換先の Toon Standard が持つ内蔵マットキャップへそのまま引き継がれる(設定不要)。NonToon も従来どおり近似変換され(乗算/加算の2枠を Toon Standard の1枠へ近似)、VRM MToon の加算スフィア(SphereAdd)も加算マットキャップとして引き継がれる。Poiyomi のマットキャップは自動では引き継がれない(変換レポートに表示。金属パーツはマテリアル別指定「MatCap Lit(金属向け)」でマットキャップ画像を引き継げる)
   - この自動引き継ぎは Quest 変換設定の詳細設定「マットキャップを引き継ぐ」でオン/オフできる(既定オン)。オフにすると引き継がれない(削除相当)ため、変換後に映り込みが乱れる場合はオフを推奨する
   - 引き継がれるもの: マットキャップ画像 / 合成モード(乗算はそのまま乗算、通常・加算・スクリーンは加算で近似)/ 強さ / マスク
   - 引き継がれないもの: 2ndマットキャップ(2枚目)、カスタム法線・視差(パララックス)・カスタムUVなどの高度なオプション。これらは変換レポートに警告が出る
2. MatCap Lit(金属パーツ向けの変換先)
   - マテリアル設定の変換方法で、変換先に VRChat/Mobile/MatCap Lit を選べる。メインテクスチャ × マットキャップの乗算にライティングを足したシンプルなシェーダーで、金属のツルッとした映り込みに最適
   - 制約: 不透明のみ / ノーマルマップ・影ランプなし / マットキャップの性質上アトラス統合の対象外(マテリアルスロットを1つ使う)
   - 使い分けの目安: 通常は自動引き継ぎ(Toon Standard)で十分。より強い金属感が欲しいパーツだけ MatCap Lit を選ぶ
3. アトラス統合との関係
   - 異なるマットキャップを持つマテリアル同士は、同じアトラスに統合されない(見た目を守るための仕様)。マットキャップを使うマテリアルはスロットを1つ使うことを見込んでおくとよい

## Meshia(ポリゴン削減の実体)について

Meshia Mesh Simplification(Ram.Type-0 氏 / MIT)は、NDMF でビルド時(Play / アップロード)にメッシュを高品質に簡略化するツールです。1.6.0 から、本ツールのポリゴン削減はこの Meshia へ連携する方式になりました。「ポリゴン削減(Meshia連携)」で目標三角形数を指定すると、本ツールが「_Quest」複製の子オブジェクトへ Meshia のアバター単位コンポーネント(Meshia Cascading Avatar Mesh Simplifier)を追加・設定します(この自動付与には Modular Avatar が必要です。未導入時はパネルに導入案内が出るだけで付与されません)。導入は VPM リポジトリ https://ramtype0.github.io/VpmRepository/ を VCC / ALCOM に追加して行います(公式ドキュメント https://ramtype0.github.io/Meshia.MeshSimplification/ )。

使い方のパターン。

- ワンボタン運用(推奨): 目標三角形数を指定して「_Quest 複製へ Meshia を追加」を押すだけ。顔・髪を含む配分は本ツールが目標へ合わせて設定します。追加後に Meshia コンポーネント側で個別の微調整もできます。
- レンダラー単位で追い込む: 崩れやすい/見せ場の部位は、追加された Meshia Cascading Avatar Mesh Simplifier で部位ごとに目標三角形数や保護設定を調整します(さらに細かく詰めたいときは、レンダラー単位の Meshia Mesh Simplifier を手動で足す方法もあります)。EditMode プレビューで品質を確認できます。
- 二重指定に注意: 同一レンダラーに、本ツールが追加した Meshia と手動で足した Meshia の両方を効かせないでください(レンダラー単位でどちらか一方にする)。両方が同じ目標へ強くかかると、ビルド後に削りすぎになります。

注意(ビルド時反映): Meshia の削減はビルド時に反映されるため、エディタ/診断のポリゴン数には出ません(EditMode プレビューはありますが、実メッシュの差し替えはビルド時です)。最終的な三角形数は即時実測(1.4.0 以降)または実測レポート(▶️ Play / SDK ビルド)で確認してください。ポリゴン削減パネルは、手動で追加された Meshia コンポーネントを検出すると二重指定の注意を表示します。

謝辞: Meshia Mesh Simplification(MIT License / Ram.Type-0 氏)に感謝します。

## ダウンロードサイズ(圧縮後10MB・展開後40MB)を超える場合の削減手順

効果が大きい順。サイズ診断の削減提案にも同じ内容が具体的な数値付きで出る。アップロード上限は圧縮後10MBと展開後40MBの2つが独立にかかり、どちらを超えてもブロックされる。展開後(非圧縮)はメッシュ・ブレンドシェイプの寄与が大きく、テクスチャ縮小だけでは下がりきらないことがあるため、その場合は下記3〜5(Quest除外・メッシュ削減・ブレンドシェイプ整理)を併用する。

1. テクスチャの Android 最大サイズを下げる(最も効く)
   - まず全テクスチャを 1024px 制限にする。それでも超えるなら、顔・体以外(小物、衣装、マスク類)を 512px にする
2. ASTC のブロックサイズを大きくする(6x6 → 8x8 で約半分になる。画質はやや落ちる)
3. 使わないオブジェクトを Quest除外(EditorOnly)にする(予備衣装、ギミック等)
4. AAO の Trace and Optimize を有効にしておく(未使用ブレンドシェイプの固定などがビルド時に効く)
5. それでも超える場合はメッシュ側の削減を検討する(ポリゴン削減(Meshia連携)で目標三角形数を指定し、ビルド時に Meshia で簡略化する / 不要ブレンドシェイプの削除)

## Quest / iOS の主な制限値(参考)

iOS は Quest(Android)と同じモバイル用の制限が適用される。

- アップロード可否はサイズのみで決まる: ダウンロードサイズ ビルド後圧縮 10MB / 展開後 40MB を超えると SDK がアップロードをブロックする(ランクでは決まらない)
- 表示: モバイルの既定表示ランクは Medium。Very Poor のアバターは既定でフォールバック表示になるが、相手は Show Avatar で表示可能(Poor も既定表示ランク超のため相手が Show Avatar しないと見えない。公式は将来この表示制限の強化を予告)
- ランク別の上限(抜粋): ポリゴン 7,500 / 10,000 / 15,000 / 20,000、スキンメッシュ 1 / 1 / 2 / 2、マテリアルスロット 1 / 1 / 2 / 4(Excellent / Good / Medium / Poor)
- PhysBone: コンポーネント 0 / 4 / 6 / 8、対象ボーン(影響 Transform)0 / 16 / 32 / 64、コライダー 0 / 4 / 8 / 16、衝突判定回数 0 / 16 / 32 / 64、コンタクト 2 / 4 / 8 / 16
- この5項目のどれか1つでも Poor 上限(Very Poor の閾値)を超えると、その分野が Very Poor になり、モバイルでは PhysBone・コンタクト・コンストレイントがすべて無効化される(揺れもの・Contact が全停止する)。コンポーネント数が 8 以内でも、他の項目が超えていれば同じく停止する

## EditorOnly と削除に関する補足

- EditorOnly タグを付けたオブジェクトはアップロードを妨げない。ビルド時に完全に除去され、サイズ・ランク計算からも外れる
- 非アクティブにするだけではビルドに含まれる。確実に外すなら EditorOnly を使う
- 手動で外す場合、アバタールートへの EditorOnly 付与と、スキンメッシュが参照するボーンの除外は避けること(本ツールの Quest除外機能はこれらを自動で検証する)
- 1.5.1 から、EditorOnly タグ付き・Quest除外に登録したオブジェクトは、診断だけでなく各セクションの一覧・プレビュー(マテリアル・衣装/トグル整理の削減量チップ・SkinnedMesh統合・PhysBone・ポリゴン削減の割り当て・隠面メッシュ候補・テクスチャなど)と変換からも自動で外れる(ビルドに含まれないため。統合前後の概算にも数えない)。非表示にした項目があるセクションには「EditorOnly/Quest除外のため n 件を非表示(ビルドに含まれないため)」を1行だけ表示する。Quest除外の登録パネルだけは全件を表示し、既に EditorOnly タグが付いているものには「EditorOnly」バッジを付ける。除外配下だけで使われるマテリアルは変換もスキップし、レポートに「EditorOnly/Quest除外配下のみで使用のため変換をスキップ: n件」と表示する(除外配下と通常の両方から使われる混在マテリアルは従来どおり変換する)

## Network ID(PC/Quest間の揺れ物の掴み同期)

PhysBone などの揺れ物の掴み・ポーズ・ストレッチ状態は Network ID をキーにネットワーク同期される。ID をアップロード時の自動採番に任せると、PC 版と Quest 版で PhysBone の数・並びが違うため別々の ID が振られ、他ユーザーから見て「掴んでいる揺れ物」がズレることがある。

- 本ツールは、元アバターを基準にした一貫した Network ID を「_Quest」複製へ変換時に書き込む(既定はオン)。オン/オフの切り替えは、単体の Quest 対応ツール(QuestConverterWindow)の詳細設定「Network IDを割り当てる」、および単体の PC 軽量化ツール(PCOptimizerWindow)の実行セクションにあります。統合ウィンドウ(Avatar Studio)では常にオンで、詳細設定にはこの項目を出していません。PC 軽量化も同じ基準で「_Opt」複製へ ID を書き込むため、同じ論理 PhysBone が PC/Quest の両方で同じ ID になる
- 元アバターに既に Network ID(SDK の「Network ID Utility」等で割り当て済み)がある場合は、そのオブジェクトの ID をそのまま引き継ぐ。未割り当てのものだけ SDK の採番規則(最小 ID 10)に従って新規付与する
- 元アバターは一切変更せず、複製の Network ID コレクションのみを書き換える。削除された PhysBone のエントリは書かれない
- PhysBone のマージ先(共通親に新設されるコンポーネント)など、元アバターに対応が無いオブジェクトはベストエフォートで新規 ID を付与する。マージ構成は PC/Quest で異なり得るため、マージ先の ID は完全な相互一致を保証しない(掴み対象になりにくいマージ済みチェーンでは実害は小さい)
- 同じ揺れ物を PC/Quest で一致させたい場合は、両ツールを同じ元アバターにかけること(ID テーブルは元アバター基準のため)

## 制限・注意

- Quest の仕様上、メッシュの透過は Particles 系シェーダーのみ。Toon Standard / Toon Lit に透明・カットアウトはない
- メッシュの半透明は Particles Additive/Multiply による近似のみ。VRChat/Mobile/Particles/Alpha Blended はアバター許可リスト外のため使用不可(実機で不透明フォールバックされる)
- アウトライン、ラメ、ファーなどの高度な質感はベイク対象外(変換レポートに警告が出る)。マットキャップは Toon Standard の内蔵マットキャップへ引き継がれる(「マットキャップ」セクション参照。2ndマットキャップ・カスタムUV/視差などの高度なオプションは対象外)
- 前髪影(疑似影 / FakeShadow)やアウトライン専用(OutlineOnly)などの効果専用シェーダーは、Quest では再現できず不透明の白い面としてベイクされてしまうため、自動で非表示化される(髪などのベース面はそのまま変換される。個別に別の変換方法を指定した場合はそちらが優先される)
- ポリゴン削減(Meshia連携)は Meshia がビルド時に簡略化するため、削減結果はエディタ診断ではなく即時実測/実測レポートで確認する。目標を極端に下げると形状が崩れるため、まずは無料の削減(隠面メッシュ削減・衣装整理・Quest除外)を先に済ませ、崩れやすい部位は Meshia コンポーネント側でレンダラー単位に追い込むと崩れにくい。元アバターは変更されず、Meshia は複製にのみ追加される
- リムライトは既定では変換されない(挙動の違いで想定外のハイライトが出るため)。必要なら変換設定の「リムライトを近似変換」をオン
- エミッションマップ未設定のマテリアルはエミッションなしとして変換される(単色発光を使いたい場合は生成マテリアルで手動設定)
- 診断のランクはエディタ上の状態で計算されるため、AAO 等がビルド時に統合する分は反映されず、実際より悪めに出ることがある。サイズ診断も推定値なので、最終確認は SDK のビルド結果、または RARA/実測レポート(_Quest 複製を ▶️(Play)、あるいは SDK の Build & Test / Upload で最終複製の実測値・実バンドルサイズを確認)で行うこと。1.4.0 からは、変換完了時に自動で「ビルド不要の実測」も走る(NDMF の手動ベイク API でビルド時と同じ MA/AAO 等の処理を一時コピーへ適用して数秒で測定し、一時コピーと生成アセットは自動削除。実測レポートウィンドウの「今すぐ実測(ビルド不要)」ボタンや設定でオフにもできる=既定オン)。ただし即時実測でもダウンロードサイズ(10MB 判定)の実ファイルサイズは推定表示のままで、lilToon 等の SDK ビルド固有処理も含まれない(ランク・ポリゴン等の計算はビルド時実測と同一。NDMF 未導入時はスキップ)。実ファイルサイズや最終的な見え方は SDK の Build & Test / Upload で確認すること
- 再変換は生成アセットを GUID 保持で上書きするため参照は壊れない(アニメーターコントローラーのみ作り直しになるので、旧 _Quest 複製は新しいものへ置き換え推奨)
- Quest除外は相対パスで記録するため、同名の兄弟オブジェクトや名前に「/」を含むオブジェクトは登録できない(登録時に検証される)

## トラブルシューティング

- 髪が Quest 版で消えた: 旧バージョンの透過自動非表示が原因。最新版では大型メッシュ・髪は自動非表示の対象外になったため、再変換すれば直る
- 変換後に一部が真っ白に光る: エミッション設定の引き継ぎが原因のことが多い。最新版では自動で対処されるため、再変換すれば直る。残る場合は該当マテリアルの Emission を生成マテリアルで手動調整
- メニューで切り替える目・衣装が Quest で消える: 最新版ではメニュー/ギミック参照のマテリアルも自動変換される。マテリアル設定に「メニュー/ギミック参照」バッジ付きで表示されているか確認
- チーク等が顔に板として乗る: 既定は「透過マテリアルの既定処理 = 自動で半透明を再現」で乗算/加算に近似される。板に見える場合は、マテリアル設定でその既定処理(非表示にする 等)や個別のマテリアル指定を確認する
- SDK ビルド画面に赤いシェーダーアラート(Mobile 非対応シェーダー)が大量に出てビルドできない: ビルドターゲットが Android なのに、SDK パネルで PC 用アバター(PC 用シェーダーのままの「_Opt」や変換前の元アバター)を選んでいるのが原因。アップロードするのは「_Quest」の方。アップロード手順ガイドの手順④「_Questだけをアクティブ化」を押すと、「_Quest」だけがシーンでアクティブになり(元アバターと「_Opt」は非アクティブになる)、SDK パネルで正しい複製を選びやすくなる。ガイドはビルドターゲットと各複製のアクティブ状態が食い違うと警告も表示する。あわせて SDK の「Multi-Platform Build & Publish」(複数プラットフォーム同時ビルド)は使わない(1つずつプラットフォームを選ぶ)
- 暗いワールドで真っ黒になる: 最新版では lilToon の最低明るさを引き継ぐため再変換で改善する
- 変換したのにパフォーマンスが改善しない / PhysBone が減っていない: 変換で作った新しい「_Quest」複製ではなく、前回変換した古い複製をビルド/アップロードしている可能性が高い。最新版は再変換時に古い複製を自動削除して新しい複製をアクティブ化・ハイライト表示するので、再変換してハイライトされた複製をアップロードすること。旧版で作った古い複製がシーンに残っている場合は手動で削除する(SDK パネルの「Previous build of ...」は前回ビルドの数値なので、必ず新しい複製を選び直して確認する)
